Pretendard

(1) 스타일 아키텍처 — SCSS/SASS

  • SCSS 구조 설계 전략

  • 디자인 토큰 중앙화 — _variables.scss
    색상·간격·폰트 등 디자인 시스템의 핵심 값을 _variables.scss에 CSS Custom Properties로 정의해 단일 출처(Single Source of Truth)를 유지했습니다. 토큰 변경 시 한 파일만 수정하면 프로젝트 전체에 즉시 반영됩니다.
  • 컴포넌트 단위 파일 분리 — @use / @forward
    SCSS의 @use와 @forward로 컴포넌트별 스타일 파일을 모듈화해 전역 네임스페이스 오염을 방지하고, 관심사 분리를 명확히 했습니다.
  • 믹스인 기반 반응형 처리
    미디어 쿼리를 믹스인으로 추상화해 반복되는 브레이크포인트 코드를 제거하고, 새로운 컴포넌트 추가 시 일관된 반응형 패턴을 빠르게 적용할 수 있도록 했습니다.

(2) Firebase 아키텍처 — Auth · Firestore · Realtime

Firebase Auth — 이메일/비밀번호 방식으로 가게 사장님 회원가입·로그인·로그아웃을 처리하고, onAuthStateChanged로 로그인 상태를 앱 전역에서 실시간 감지합니다.
Firestore (stores 컬렉션) — 가게 등록·수정·삭제와 두쫀쿠 재고(duzzonCount) 업데이트, 오늘 마감 여부(closedDate) 토글을 CRUD로 관리하며, onSnapshot으로 지도 화면에 실시간 반영합니다.
권한 흐름 — 로그인한 사용자만 가게 등록/수정/삭제가 가능하고, 비로그인 사용자는 지도에서 가게 목록 조회만 할 수 있는 역할 기반 접근 제어(RBAC) 구조입니다.

(3) 실시간 재고 동기화 — Firestore onSnapshot

Firestore의 onSnapshot 리스너를 지도 컴포넌트 마운트 시점에 등록해, 사장님이 재고를 수정하는 즉시 페이지 새로고침 없이 지도 마커에 반영되는 실시간 동기화를 구현했습니다.

마커는 재고 수량에 따라 초록(구매가능) / 노랑(마감임박) / 빨강(품절) 세 가지 상태로 즉시 시각화되어, 사용자가 별도 인터랙션 없이 한눈에 재고 상황을 파악할 수 있습니다. 컴포넌트 언마운트 시 리스너를 정리(cleanup)해 메모리 누수를 방지했습니다.

(4) 역할 기반 UI 분기 — 사장님 / 손님 권한 분리

Firebase Auth의 로그인 상태와 Firestore 문서의 소유자(uid)를 클라이언트에서 대조해, 로그인한 사장님에게만 재고 수정·오늘 마감 토글·매장 삭제 버튼을 노출하고 일반 사용자에게는 전화하기 버튼만 렌더링하는 조건부 UI를 구현했습니다.

매장 등록 흐름은 로그인/회원가입 → 매장 정보 입력 → 카카오 지도 위치 지정을 단일 페이지에서 스텝 형태로 처리해, 외부 서비스 연동이 포함된 복잡한 폼을 UX 부담 없이 완료할 수 있도록 설계했습니다.

(5) 복합 조건 영업 상태 판단 로직

단순 재고 여부를 넘어 영업 요일 · 영업시간 · 오늘 마감(closedDate) 세 가지 조건을 JavaScript Date 기반으로 조합해 현재 시점 영업 여부를 클라이언트에서 계산하는 순수 함수를 구현했습니다.

재고 필터(전체 / 구매가능 / 마감임박 / 품절)와 "영업중" 필터를 독립 상태로 관리하고 교집합으로 최종 리스트를 도출해, 재고가 있어도 폐점한 가게를 걸러낼 수 있는 실용적인 사용자 경험을 제공합니다.

(6) 범용 재고 지도 서비스 — 확장 가능한 아키텍처

지도·인증·재고 관리 로직을 특정 디저트에 종속되지 않도록 설계해, 아이콘과 브랜드명만 교체하면 동일한 서비스를 어떤 한정 수량 판매 아이템에도 즉시 적용할 수 있는 범용 재고 현황 지도 서비스 구조를 구축했습니다.

단기 프로젝트(5일)임에도 재사용성과 확장성을 의도적으로 고려해 설계한 점이 이 프로젝트의 기술적 강점입니다.
